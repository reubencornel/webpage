1 A practical approach to programming with assertions - David Rosenblum    SOFTWARE_ENGINEERING ASSERTIONS

This is a paper that describes a system developed by the author that helps them specify assertions. They also provide a classification of the types of assertions that they used in the code.

As a side note I must say that though instructive about the type of assertions, I find it disappointing the authors leave out details about assertions from a program verification perspective.

1.1 General notes on Assertions

  • Assertions describe what a system is supposed to do rather than how
  • Assertions find use is run time in
    • Testing
    • Debugging
    • Maintainance
  • Viewed as a permanent part of programming defensively.
  • Assertions state the properties of the function to any one who reads the code.
  • The reasons why assertions had not seen large scale use is because
    • It did not seem to usable to the average developer
      • The tools did not integrate
        • Did not provide the ability to customize both the types of assertions and the behaviours
    • Lack of studies related to what types of checks that can be done with assertions.
  • Interesting side note Most software faults are interface faults.
    • That implies that designing the types of any system is particularly important.
  • Assertions should avoid having side effects
    • Reason being assertions should simply check the computation rather than be an active part of it.

1.2 Classification of assertions

1.2.1 Specification of function interfaces.

They describe function behaviour at a level of abstraction as seen by the callers

1.2.1.1 Consistency between arguments

Assertions that are used to describe the constraints between arguments of a functions. As a result, we can use them to validate values of parameters passed into the function.

For example the paper describes a function that stores a sequence of tokens, the assertion there describes the relation between the type of token and the value of the actual token.

1.2.1.2 Dependency of return value on arguments

It describes the type of relationship between the input parameters and the return value of the function.

1.2.1.3 Effect on global state

Used to describe the effect of a function on variables that store global state. For example, they describe the state of the variable once the function execution is almost complete.

1.2.1.4 Context within which a function is called

Used to describe the state of the system when the function is called.

1.2.1.5 Frame specifications

Used to specify that a function does not manipulate the global state or the contents of a variable passed in through a reference.

1.2.1.6 Sub range membership of data

Validate if a sub range of data passed into a function meets the required criteria.

1.4 Conclusion

The mechanism for assertions mentioned here has a number of interesting things to note

  • The assertions are mentioned as comments and are inserted into the code using a preprocessor. The advantage of doing this is that we can have separate copies of the binary built with out assertions.
  • Assertions have a severity level assigned to assertion failures
  • A flexible way of dealing with assertion failures, which could be based on the severity level. This to a large extent kind of reminds me of using some sort of exeception handling capability, but I have not thought about it too much.

The paper does address the fact that it does not cover assertions from the point of view of verification, with no valid reason.
